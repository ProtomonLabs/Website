Variant annotation is a critical layer in any genomic analysis pipeline. It bridges the gap between raw variant calls and biological interpretation — but doing it at scale requires infrastructure, not just a script.

## Why Infrastructure Matters

A single VCF file is manageable. A thousand are not. When you're processing population-scale sequencing data, annotation becomes an infrastructure problem:

- **Throughput**: Can your pipeline handle 10,000 genomes in a week?
- **Consistency**: Are annotations reproducible across batches and tool versions?
- **Extensibility**: How do you add a new annotation source without rebuilding everything?
- **Portability**: Does your pipeline run on local clusters, cloud, and HPC?

## Infrastructure Stack

Here's the approach we use for production-grade annotation:

### 1. Workflow Management

```bash
# Nextflow pipeline for scalable annotation
nextflow run protomon/annotate \\
  --input s3://project/callsets/ \\
  --output s3://project/annotated/ \\
  --reference GRCh38 \\
  --cache /vep-cache \\
  -profile docker \\
  -with-report annotation-report.html
```

Use workflow managers (Nextflow, WDL/Cromwell) that handle task scheduling, retry logic, and resource management. Don't orchestrate annotation with shell scripts.

### 2. Layered Annotation Architecture

```
Raw VCF
  └─ Layer 1: Core annotation (VEP)
       └─ Layer 2: Conservation (phyloP, phastCons, CADD)
            └─ Layer 3: Frequencies (gnomAD, TopMed, local)
                 └─ Layer 4: Regulatory (Ensembl Regulatory, Roadmap)
                      └─ Layer 5: Tissue-specific (GTEx, ENCODE)
                           └─ Layer 6: Custom (project-specific)
```

Each layer is independent and cacheable. Add or remove layers without disrupting the pipeline.

### 3. Annotation Database Management

```python
# pseudocode for annotation database versioning
annotation_store = AnnotationStore("s3://annotations/")
annotation_store.register("vep", version="114", build="GRCh38")
annotation_store.register("cadd", version="1.7", build="GRCh38")
annotation_store.register("gnomad", version="4.1", build="GRCh38")

# Reproduce any past analysis
pipeline = annotation_store.reproduce(timestamp="2026-06-01")
```

### 4. Standardized Output Format

Define a schema for annotated variants once, across all pipelines:

```json
{
  "variant_id": "1-1000000-G-A",
  "consequence": "missense_variant",
  "gene": "GENE1",
  "impact": "MODERATE",
  "cadd_score": 18.5,
  "gnomad_af": 0.0023,
  "regulatory_features": ["enhancer_heart"]
}
```

## Key Decisions

- **Use the same reference build throughout**: GRCh38 or T2T-CHM13, but pick one and standardize
- **Cache aggressively**: Local VEP caches reduce runtime by 10x
- **Version everything**: Annotation databases change. Tag every release.
- **Design for incremental updates**: New samples should annotate in minutes, not days
